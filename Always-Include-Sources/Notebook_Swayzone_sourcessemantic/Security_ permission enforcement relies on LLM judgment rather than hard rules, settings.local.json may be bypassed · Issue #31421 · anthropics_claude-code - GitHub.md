#### This nails the core problem:

instruction-based security is not security

. An LLM that "decides" it has permission is functionally identical to no permission system at all.

The gap is between

guardrails (system prompt instructions) and

guardrails (code-level enforcement). The

settings.local.json

allow list should be a hard boundary, but as reported, the model can reason around it.

#### Two complementary approaches that help here:

1. PreToolUse hooks for deterministic enforcement

#### Claude Code's hook system can enforce rules outside the model's reasoning loop:

{
  "hooks": {
    "PreToolUse": \[{
      "matcher": "Bash",
      "command": "python -c \"import sys,json; args=json.loads(sys.stdin.read()); cmd=args.get('tool\_input',{}).get('command',''); blocked=\['git push','git commit','rm -rf'\]; sys.exit(2) if any(b in cmd for b in blocked) else sys.exit(0)\""
    }\]
  }
}

Exit code 2 = hard block, model can't override. This moves enforcement from "the model follows instructions" to "the code rejects the call."

2. Content-aware scanning on tool arguments

Beyond simple string matching, scanning tool arguments for risk patterns catches cases where the model uses alternative commands to achieve the same effect (e.g.,

echo '...' > file

instead of the Write tool). We built a deterministic scanner that classifies risk in ~0.05ms — fast enough to run on every tool call without perceptible latency:

sentinel-ai

https://github.com/MaxwellCalkin/sentinel-ai

The fundamental fix is what you said: hard-enforce at the tool-call level, independent of LLM reasoning. Hooks are the mechanism that exists today to do this.

allowed-tools

also lacks hard enforcement.
