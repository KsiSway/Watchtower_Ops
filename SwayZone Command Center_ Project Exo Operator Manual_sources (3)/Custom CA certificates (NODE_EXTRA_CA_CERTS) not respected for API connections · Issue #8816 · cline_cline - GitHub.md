Plugin Type

VSCode Extension

#### Cline Version

3.52.0, 3.37.1, and 3.51.0 with VSCode 1.107.1

What happened?

Users in enterprise with custom CA certificates experience TLS certificate verification failures when connecting to OpenAI-compatible model services. The issue manifests as SSL/TLS errors when the model service endpoints use certificates signed by a custom Certificate Authority that is in the user's trust bundle.

Connections to OpenAI-compatible endpoints fail with certificate errors

The only workaround is setting NODE\_TLS\_REJECT\_UNAUTHORIZED='0' before launching VS Code (which is insecure)

Older versions of Cline worked correctly with the same model services and trust bundle

Steps to reproduce

Users in enterprise environments should set the NODE\_EXTRA\_CA\_CERTS environment variable before launching VS Code:

#### Windows (PowerShell):

$env:NODE\_EXTRA\_CA\_CERTS="C:\path\to\ca-bundle.crt"

#### Windows (Command Prompt):

set NODE\_EXTRA\_CA\_CERTS=C:\path\to\ca-bundle.crt

macOS/Linux:

export NODE\_EXTRA\_CA\_CERTS=/path/to/ca-bundle.pem

For permanent configuration, add the environment variable to your shell profile or system environment variables.

Set up an OpenAI-compatible endpoint with a certificate signed by a custom CA

Add the CA certificate to a PEM file

Set NODE\_EXTRA\_CA\_CERTS to point to that PEM file

Launch VS Code and configure Cline to use the OpenAI-compatible endpoint

Verify that API calls fails (or succeeds after fix) without requiring NODE\_TLS\_REJECT\_UNAUTHORIZED='0'

#### Root Cause according to Claude 4.5:

#### The issue was introduced by two changes:

OpenAI SDK upgrade from v4 to v6 (commit

https://github.com/cline/cline/commit/9e35048db2f8273f95a4f05fafde86d53b0ae78b

): The OpenAI Node.js SDK switched from node-fetch with httpAgent support to the Web fetch API. The old httpAgent approach respected Node.js TLS settings including NODE\_EXTRA\_CA\_CERTS. The new fetch API uses undici internally but doesn't automatically pick up custom CA certificates in all scenarios.

Proxy support patch (commit

https://github.com/cline/cline/commit/b4ea1a17a5867674b72b872645acccefbdb923c4

): Added undici for proxy support in standalone mode (JetBrains/CLI), but didn't include TLS certificate configuration.

Thanks for the detailed report! This is a well-documented issue pattern that was addressed in v3.45.0, but I want to clarify the expected behavior based on your version (3.52.0).

How Certificate Trust Works in Cline

For VS Code, Cline uses VS Code's native

globalThis.fetch

which automatically picks up certificate configuration from VS Code itself. The

NODE\_EXTRA\_CA\_CERTS

environment variable works differently in VS Code vs CLI/JetBrains:

: Relies on VS Code's built-in proxy and certificate handling. VS Code should pull trusted certificates from the OS trust store automatically.

CLI/JetBrains

: Uses undici's fetch with explicit proxy support, and

NODE\_EXTRA\_CA\_CERTS

is respected by Node.js.

Recommended Workaround

Since you're on VS Code (v3.52.0), try enabling

"Experimental: System Certificates V2"

in VS Code settings:

Open VS Code Settings (

Search for "System Certificates V2"

HTTP: Experimental: System Certificates V2

Restart VS Code

This was confirmed to fix the issue in

https://github.com/cline/cline/issues/8123

for users in similar corporate proxy/custom CA environments.

There was a regression in v3.41.0-v3.44.x where VS Code incorrectly used undici's fetch instead of VS Code's native fetch (fixed in PR

https://github.com/cline/cline/pull/8098

). Since you're on v3.52.0, that fix is included. However, VS Code's native fetch handling of custom CAs can still require the experimental setting above.

#### If the workaround doesn't help, it would help maintainers to see:

The specific error message from the Output panel (select "Cline" from dropdown)

Whether adding your CA to the Windows/macOS/Linux system trust store (not just setting the env var) makes a difference

A redacted example of your endpoint URL format (http vs https, etc.)

For more details on proxy/certificate configuration, see the

Networking and Proxies documentation

https://docs.cline.bot/troubleshooting/networking-and-proxies

I'll try the workarounds. Honestly, I'm not sure the exact same problem exists in 3.5.2. I put Claude to work on the latest version available at home based on the symptom I experience on what is likely a slightly lower version available at work. Claude observed a problem in 3.5.2 that would result in the described symptom and suggested a fix here:

main...dstaulcu:cline:mtlsfix

https://github.com/cline/cline/compare/main...dstaulcu:cline:mtlsfix

I think there's a gap in the current VS Code mode handling. While VS Code's native globalThis.fetch should respect the OS trust store, relying on the experimental "System Certificates V2" setting isn't ideal for a few reasons:

NODE\_EXTRA\_CA\_CERTS is a standard Node.js mechanism - users reasonably expect it to work without additional configuration

Experimental settings are fragile - they may change or be removed in future VS Code versions

OS trust store access may not be possible - in some corporate environments, users can set environment variables but can't modify the system trust store

The fix in my branch takes a more explicit approach: when NODE\_EXTRA\_CA\_CERTS is set in VS Code mode, using undici directly with the CA certificates passed explicitly, rather than relying on VS Code's native fetch behavior.

This ensures NODE\_EXTRA\_CA\_CERTS works predictably across all modes (VS Code, JetBrains, CLI) without requiring users to enable experimental settings or modify their OS trust store.
