Tool recommendations

https://packaging.python.org/en/latest/guides/tool-recommendations/#tool-recommendations

The Python packaging landscape consists of many different tools. For many tasks, the

Python Packaging Authority

https://packaging.python.org/en/latest/glossary/#term-Python-Packaging-Authority-PyPA

(PyPA, the working group which encompasses many packaging tools and maintains this guide) purposefully does not make a blanket recommendation; for example, the reason there are many build backends is that the landscape was opened up in order to enable the development of new backends serving certain users' needs better than the previously unique backend, setuptools. This guide does point to some tools that are widely recognized, and also makes some recommendations of tools that you should

use because they are deprecated or insecure.

Virtual environments

https://packaging.python.org/en/latest/guides/tool-recommendations/#virtual-environments

The standard tools to create and use virtual environments manually are

https://packaging.python.org/en/latest/key\_projects/#virtualenv

(PyPA project) and

https://docs.python.org/3/library/venv.html

(part of the Python standard library, though missing some features of virtualenv).

Installing packages

https://packaging.python.org/en/latest/guides/tool-recommendations/#installing-packages

https://packaging.python.org/en/latest/key\_projects/#pip

is the standard tool to install packages from

https://packaging.python.org/en/latest/glossary/#term-Python-Package-Index-PyPI

. You may want to read pip's recommendations for

secure installs

https://pip.pypa.io/en/latest/topics/secure-installs/

. Pip is available by default in most Python installations through the standard library package

https://docs.python.org/3/library/ensurepip.html

Alternatively, consider

https://packaging.python.org/en/latest/key\_projects/#pipx

for the specific use case of installing Python applications that are distributed through PyPI and run from the command line. Pipx is a wrapper around pip and venv that installs each application into a dedicated virtual environment. This avoids conflicts between the dependencies of different applications, and also with system-wide applications making use of the same Python interpreter (especially on Linux).

For scientific software specifically, consider

https://packaging.python.org/en/latest/key\_projects/#conda

https://packaging.python.org/en/latest/key\_projects/#spack

Write a “pip vs. Conda” comparison, here or in a new discussion.

easy\_install

https://packaging.python.org/en/latest/key\_projects/#setuptools

), which is deprecated in favor of pip (see

pip vs easy\_install

https://packaging.python.org/en/latest/discussions/pip-vs-easy-install/#pip-vs-easy-install

for details). Likewise, do

python setup.py install

python setup.py develop

, which are also deprecated (see

Is setup.py deprecated?

https://packaging.python.org/en/latest/discussions/setup-py-deprecated/#setup-py-deprecated

for background and

How to modernize a setup.py based project?

https://packaging.python.org/en/latest/guides/modernize-setup-py-project/#modernize-setup-py-project

for migration advice).

https://packaging.python.org/en/latest/guides/tool-recommendations/#lock-files

https://packaging.python.org/en/latest/key\_projects/#pip-tools

https://packaging.python.org/en/latest/key\_projects/#pipenv

are two recognized tools to create lock files, which contain the exact versions of all packages installed into an environment, for reproducibility purposes.

Build backends

https://packaging.python.org/en/latest/guides/tool-recommendations/#build-backends

Please, remember: this document does not seek to steer the reader towards a particular tool, only to enumerate common tools. Different use cases often need specialized workflows.

build backends

https://packaging.python.org/en/latest/glossary/#term-Build-Backend

for pure-Python packages include, in alphabetical order:

https://flit.pypa.io/en/stable/pyproject\_toml.html

– developed with but separate from

https://packaging.python.org/en/latest/key\_projects/#flit

. A minimal and opinionated build backend. It does not support plugins.

https://pypi.org/project/hatchling/

– developed with but separate from

https://packaging.python.org/en/latest/key\_projects/#hatch

. Supports plugins.

PDM-backend

https://backend.pdm-project.org/

– developed with but separate from

https://packaging.python.org/en/latest/key\_projects/#pdm

. Supports plugins.

Poetry-core

https://pypi.org/project/poetry-core/

– developed with but separate from

https://packaging.python.org/en/latest/key\_projects/#poetry

. Supports plugins. Unlike other backends on this list, Poetry-core does not support the standard

\[project\] table

https://packaging.python.org/en/latest/guides/writing-pyproject-toml/#writing-pyproject-toml

(it uses a different format, in the

\[tool.poetry\]

https://packaging.python.org/en/latest/key\_projects/#setuptools

, which used to be the only build backend. Supports plugins. Caution If you use setuptools, please be aware that some features that predate standardisation efforts are now deprecated and only

temporarily kept

for compatibility. In particular, do

python setup.py

invocations. On the other hand, configuring setuptools with a

file is still fully supported, although it is recommended to use the modern

\[project\] table in pyproject.toml

https://packaging.python.org/en/latest/guides/writing-pyproject-toml/#writing-pyproject-toml

) whenever possible and keep

only if programmatic configuration is needed. See

Is setup.py deprecated?

https://packaging.python.org/en/latest/discussions/setup-py-deprecated/#setup-py-deprecated

. Other examples of deprecated features you should

use include the

setup\_requires

argument to

\[build-system\] table

https://packaging.python.org/en/latest/guides/writing-pyproject-toml/#pyproject-guide-build-system-table

pyproject.toml

instead), and the

easy\_install

command (cf.

pip vs easy\_install

https://packaging.python.org/en/latest/discussions/pip-vs-easy-install/#pip-vs-easy-install

https://packaging.python.org/en/latest/key\_projects/#distutils

, which is deprecated, and has been removed from the standard library in Python 3.12, although it still remains available from setuptools.

Build backends for extension modules

https://packaging.python.org/en/latest/guides/tool-recommendations/#build-backends-for-extension-modules

For packages with

extension modules

https://packaging.python.org/en/latest/glossary/#term-Extension-Module

, it is best to use a build system with dedicated support for the language the extension is written in, for example:

https://packaging.python.org/en/latest/key\_projects/#setuptools

– natively supports C and C++ (with third-party plugins for Go and Rust),

meson-python

https://packaging.python.org/en/latest/key\_projects/#meson-python

– C, C++, Fortran, Rust, and other languages supported by Meson,

scikit-build-core

https://packaging.python.org/en/latest/key\_projects/#scikit-build-core

– C, C++, Fortran, and other languages supported by CMake,

https://packaging.python.org/en/latest/key\_projects/#maturin

– Rust, via Cargo.

Building distributions

https://packaging.python.org/en/latest/guides/tool-recommendations/#building-distributions

The standard tool to build

source distributions

https://packaging.python.org/en/latest/glossary/#term-Source-Distribution-or-sdist

https://packaging.python.org/en/latest/glossary/#term-Wheel

for uploading to PyPI is

https://packaging.python.org/en/latest/key\_projects/#build

. It will invoke whichever build backend you

https://packaging.python.org/en/latest/guides/writing-pyproject-toml/#pyproject-guide-build-system-table

pyproject.toml

python setup.py sdist

python setup.py bdist\_wheel

for this task. All direct invocations of

https://packaging.python.org/en/latest/discussions/setup-py-deprecated/#setup-py-deprecated

If you have

extension modules

https://packaging.python.org/en/latest/glossary/#term-Extension-Module

and want to distribute wheels for multiple platforms, use

cibuildwheel

https://packaging.python.org/en/latest/key\_projects/#cibuildwheel

as part of your CI setup to build distributable wheels.

Uploading to PyPI

https://packaging.python.org/en/latest/guides/tool-recommendations/#uploading-to-pypi

For projects hosted on or published via supported CI/CD platforms, it is recommended to use the

Trusted Publishing

https://packaging.python.org/en/latest/guides/publishing-package-distribution-releases-using-github-actions-ci-cd-workflows/#trusted-publishing

, which allows the package to be securely uploaded to PyPI from a CI/CD workflow without a manually configured API token.

As of November 2024, PyPI supports the following platforms as Trusted Publishing providers:

GitHub Actions (on

https://github.com

GitLab CI/CD (on

https://gitlab.com

ActiveState

Google Cloud

The other available method is to upload the package manually using

https://packaging.python.org/en/latest/key\_projects/#twine

python setup.py upload

for this task. In addition to being

https://packaging.python.org/en/latest/discussions/setup-py-deprecated/#setup-py-deprecated

, it is insecure.

Workflow tools

https://packaging.python.org/en/latest/guides/tool-recommendations/#workflow-tools

These tools are environment managers that automatically manage virtual environments for a project. They also act as “task runners”, allowing you to define and invoke tasks such as running tests, compiling documentation, regenerating some files, etc. Some of them provide shortcuts for building distributions and uploading to PyPI, and some support lock files for applications. They often call the tools mentioned above under the hood. In alphabetical order:

https://packaging.python.org/en/latest/key\_projects/#flit

https://packaging.python.org/en/latest/key\_projects/#hatch

https://nox.thea.codes/en/latest/index.html

https://packaging.python.org/en/latest/key\_projects/#pdm

https://packaging.python.org/en/latest/key\_projects/#pipenv

https://packaging.python.org/en/latest/key\_projects/#poetry

https://tox.wiki/en/latest/index.html
